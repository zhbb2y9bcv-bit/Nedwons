# NedwonsMLS

The Swift package that gives the iOS app on-device MLS by wrapping the Rust `mls-core` through the
`mls-ffi` UniFFI boundary (ADR-0007). It is a **sibling** of `NedwonsKit`, not a target inside it,
so NedwonsKit's pure-Swift tests keep building with no native dependency; the app composes both.

## Layout

- `Sources/MlsFfi/mls_ffi.swift` — the **generated** UniFFI bindings (committed build artifact;
  regenerated by the build script, stale-checked in CI).
- `MlsFfi.xcframework` — the prebuilt Rust core (macOS + iOS device + iOS simulator slices). A large
  derived binary; **not committed** — build it with the script below.
- `Tests/MlsFfiBridgeTests` — the Swift↔Rust integration test (two clients exchange a real MLS
  message, persist, relaunch, retry, reject hostile input). On the host it links the macOS slice.

## Build & test

The XCFramework must exist before `swift build`/`swift test` (the package links it):

```sh
# from the repo root — builds all slices, generates bindings, assembles the xcframework
scripts/build_mls_ffi.sh

# host integration test (links the macOS slice; no simulator needed)
cd apps/ios/NedwonsMLS && swift test

# RUN the same tests in the iOS simulator (ios-arm64-simulator slice)
scripts/test_mls_sim.sh

# compile/link the device slice (compile-only — running needs a physical iPhone, R-101)
xcodebuild build -scheme NedwonsMLS -destination 'generic/platform=iOS' CODE_SIGNING_ALLOWED=NO
```

Note: the simulator run uses a derived-data path **outside** TCC-protected folders
(Desktop/Documents) — the simulator's xctest runner cannot read bundles there and fails with a
misleading "bundle does not exist" error. `scripts/test_mls_sim.sh` handles this.

## Security boundary

No MLS logic lives in Swift — every call marshals into `mls-core`. No key material, ratchet secret,
or raw group state crosses into Swift; only opaque ciphertext, key packages, welcomes, and decrypted
application plaintext do. See ADR-0007 for the full contract.
